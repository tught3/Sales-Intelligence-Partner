# Agent Workflow Rules

## GSD 기본
- 모든 작업은 GSD 우선이다.
- 새 세션에서는 `.planning/STATE.md`와 `.planning/context/ACTIVE_SUMMARY.md`를 먼저 본다.
- 작은 작업은 `gsd:fast`, 중간 작업은 `gsd:quick`, 큰 작업/다중 지시는 `gsd:plan-phase` 기준으로 분류한다.
- GSD 기준 폴더는 항상 `C:\AI-automatic-expense-tracker\.planning`만 사용한다.

## QMD 검색
- 프로젝트 문서, 과거 결정, 메모, 노트 검색은 먼저 QMD로 시도한다.
- QMD로 부족할 때만 grep/read 범위를 한 단계씩 넓힌다.

## 에이전트 계층 구조
- **메인 에이전트(총책임자)**: 사용자 지시를 해석·분해하고 작업 에이전트에 병렬 배정한다.
- **작업 에이전트**: 충돌 없는 독립 범위를 맡아 병렬 작업한다.
- **검토관리자 에이전트**: 작업 결과를 검토하고 누락/회귀/연결 경로 오류를 찾는다.
- 검토관리자 승인 전까지 완료 보고 금지.
- 메인 에이전트가 최종 확인 후 사용자에게 보고.
- 작은 작업은 메인 에이전트 단독 또는 작업 에이전트 1개 + 검토관리자로 축소 가능.
- 에이전트 수는 작업 규모에 따라 유동적으로 결정한다.

## 장시간 작업 회피
- 가장 좁고 구체적인 범위부터 시작한다.
- 근거가 없으면 관련 호출 경로나 인접 폴더로 한 단계씩만 넓힌다.
- 3분 이상 걸릴 가능성이 높으면 장시간 탐색을 계속하지 말고, 이유/더 빠른 대안/최소 단위 작업을 짧게 보고한다.

## 진행상황 보고
- 작업 시작 직후 가장 좁은 범위와 첫 확인 대상을 1~2문장으로 보고한다.
- 긴 작업은 30초 안팎마다 짧게 보고한다.
- 가능하면 `진행 20%`, `진행 60%` 식으로 퍼센트를 붙인다.
- 작은 수정 1개가 끝날 때마다 `.planning/context/ACTIVE_SUMMARY.md`와 최신 snapshot을 갱신해서, 중간에 끊겨도 바로 이어받을 수 있게 한다.

## 컨텍스트 위생
- 세션 시작 직후, 긴 작업 시작 전, 최종 보고 직전에 `node scripts/gsd-context-hygiene.mjs`를 실행한다.
- 자동 정리는 `.planning/context/snapshots` 같은 자동 산출물에만 적용한다.

## 효율 규칙
- 이미 읽은 파일 재확인 금지.
- 독립적인 도구 호출은 병렬 실행.
- 20줄↑ 탐색/분석은 서브에이전트 위임.
- 최소 출력: diff 위주 응답, 변경 없는 코드 전체 출력 금지.
- 정밀 타격: 경로 제공 시 전체 탐색 없이 직접 사용.

## 컨텍스트 보존
- 복잡한 작업(API 연동, 대규모 리팩터, 다중 파일 수정)은 서브 에이전트에 위임해 메인 컨텍스트를 보존한다.
- `ls`, `npm test`, 회귀 실행 등 터미널 출력이 긴 명령은 압축 요약만 참조한다 (전체 출력 직접 읽기 금지).
- 대량 로그 분석은 메인에서 직접 읽지 않고 서브 에이전트에 요약 위임한다.
